1//! # `kafka_event_aggregator`
2//!
3//! This program _aggregates_ events from a `_rawEvents` Kafka topic, and emits
4//! events onto an `_events` topic. It is responsible for zipping together events
5//! and metadata messages emitted by different sources into "frames".
6//!
7//! ## Operation
8//!
9//! When data arrives on `config.input_topic` (usually `_rawEvents`), `kafka_event_aggregator` will:
10//! - Check whether this data forms part of an existing frame. If not, a new Frame will be created,
11//!   with a time-to-live `config.expiry_offset_ms` in the future.
12//! - Append neutron events from the new data, or combine veto statuses from the new data, with the
13//!   existing or newly-created frame
14//!
15//! When a frame of data's time-to-live expires, and the next `config.frame_queue_poll_interval`
16//! timer interval expires, `kafka_event_aggregator` will:
17//! - Sort the events in the frame by increasing time of flight
18//! - Emit a `pu00` frame metadata message to the `config.output_topic` (usually `_events`) stream
19//! - Emit zero or more `ev44` messages, each containing at most `config.max_events_per_message`
20//!   neutron events per `ev44` message.
21//!
22//! ## `events` stream format
23//!
24//! Consumers of the `_events` stream may make the following assumptions:
25//! - Each frame is made of one `pu00` message and any number of `ev44` messages.
26//! - All messages in each frame will go to the same Kafka partition (the messages that make up
27//!   a frame all have the same Kafka message key).
28//! - Messages are produced to Kafka using an idempotent producer (enable.idempotence = true),
29//!   ensuring that retries do not create duplicate messages. Kafka guarantees in-order delivery
30//!   within each partition.
31//! - Events will be emitted ordered in time-of-flight, both within a single `ev44` message and
32//!   across multiple `ev44` messages within one frame, if `config.sort_events_by_tof = true`.
33//! - Each `ev44` on the `_events` stream will contain at most `max_events_per_message` events
34//! - The `reference_time` of all `pu00` and `ev44` messages which make up a frame will be
35//!   identical.
36//!
37//! For the avoidance of doubt, it is **not** correct to assume that:
38//! - Each frame will only contain one `ev44`. This may be true when count rate is small, but may
39//!   not be true for instruments with a high count rate.
40//! - A frame will contain an `ev44`. If all events are vetoed, no `ev44` messages will be emitted.
41//!   (A `pu00` frame metadata message will still be emitted, containing the veto flags).
42//! - Frames will be emitted in increasing order of reference time. Due to the underlying UDP
43//!   connection from the streaming hardware, this cannot be guaranteed (but is expected to be
44//!   true *most* of the time).
45//! - A frame must be "complete" to be emitted. Due to the underlying UDP connection from the
46//!   streaming hardware, and the fact that each streaming detector may emit any number of event
47//!   messages, it is not possible to tell whether all data from a frame has been
48//!   received. It is **assumed** that all data corresponding to a frame will appear in the `events`
49//!   stream within `config.expiry_offset_ms` of the *first* message for that frame arriving.
50//!   Events that come in too late are dropped.
51//!
52//! ## Consuming from the `_events` stream
53//!
54//! The recommended approach to consume from the events stream, while accounting for frame
55//! metadata (vetoes, period number, proton charge) is to keep a per-partition
56//! **array** of the most recently seen frame metadata, keyed by Kafka partition of the message.
57//! This metadata applies to all `ev44` event messages on that partition until the next `pu00`
58//! arrives.
59//!
60//! In a live consumer:
61//! - When a `pu00` (metadata) `msg` arrives: Update `array[msg.partition]` with the new metadata
62//! - When an `ev44` (event data) `msg` arrives: Use `array[msg.partition]` as the metadata for
63//!   those neutron events
64//!
65//! This ensures that each neutron event is associated with the correct metadata, even if the
66//! metadata for frame `N+1` arrives between the metadata and the events for frame `N` (which it
67//! may, iff those frames are on different partitions).
68//!
69//! This approach also works across multiple consumer processes: a consumer will receive all
70//! messages from a single frame, but different frames may be processed by different consumers.
71//!
72//! ## Metrics
73//!
74//! Prometheus-compatible metrics are provided by a scrape endpoint, configured by
75//! `config.metrics_bind_addr`. This endpoint can also be accessed by curl or a standard web
76//! browser.
77//!
78//! ## Logging
79//!
80//! `kafka_event_aggregator` accepts command-line `-v` and `-q` flags to increase and decrease
81//! logging verbosity respectively. The flags may be supplied multiple times.
82//!
83//! ## Benchmarks
84//!
85//! A benchmark of the full aggregation process is available in the `benchmarks/` folder, and
86//! is run using `cargo bench`.
87//!
88//! ## Documentation
89//!
90//! `cargo doc --no-deps --open`.
91//!
92//! ## Configuration
93//!
94//! An example config file is provided in `config_example.toml`. This may be copied to `config.toml`
95//! and adjusted to taste.
96//!
97//! The aggregator is then run with:
98//! `kafka_event_aggregator --config path/to/config.toml`
99//!
100//! ### Passing configuration to container
101//!
102//! If you are using the container and want to pass a config file as above you need to mount it as a volume at runtime.
103//! You can do this like so:
104//! `docker build . -t kae:latest && docker run -v ./config.toml:/config.toml -it kae:latest --config ./config.toml`
